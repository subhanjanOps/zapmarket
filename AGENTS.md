## Purpose

This guide gives AI coding agents the context needed to work effectively in this repository. The codebase is a Go monorepo of six microservices. Two are fully implemented; four are scaffolded stubs awaiting implementation.

## Architecture

- **gRPC** for synchronous inter-service calls
- **Kafka** for async events (`order.created`, `payment.processed`, `inventory.reserved`, `user.registered`, etc.)
- **No cross-service DB access** — cross-service data goes through gRPC or Kafka only
- Each service owns its own PostgreSQL database
- `design.md` is the authoritative reference for Kafka topics, the saga/checkout flow, Redis patterns, and data store assignments

**Ports:**

| Service | HTTP | gRPC |
|---|---|---|
| `auth-service` | 8080 | 50051 |
| `product-catalog-service` | 8081 | 50052 |

**Infrastructure:** `docker-compose.yml` currently runs only PostgreSQL. Kafka, Redis, Elasticsearch, MinIO, and Debezium configs are present but commented out.

## Service Internal Structure

All implemented services follow this layered pattern:

```
internal/
  domain/        # models and domain types
  repository/    # DB access via database/sql (no ORM); lib/pq driver
  service/       # business logic; contracts defined in repository_interfaces.go
  handler/
    http/        # HTTP handlers; base.go has shared ErrorResponse/SuccessResponse helpers
    grpc/        # gRPC server implementation
  middleware/    # auth.go: validates JWTs by calling auth-service ValidateToken RPC
  errors/        # domain-specific error types
pkg/config/      # env-var config with godotenv .env support
proto/           # .proto files + generated *pb.go (copied locally, not a shared module)
docs/            # Swagger docs (auto-generated by swaggo/swag)
```

Use `services/auth-service/` as the reference for new services. Use `services/product-catalog-service/` as the reference for the chi router + auth middleware pattern.

## Auth & RBAC Pattern

`auth-service` is the single source of truth for JWT issuance and validation. **Other services must not validate JWTs locally.** Instead, they dial `auth-service` gRPC and call `ValidateToken`, exactly as implemented in `services/product-catalog-service/internal/middleware/auth.go`.

- `AuthMiddleware.Authenticate` — extracts `Authorization: Bearer <token>` and calls `ValidateToken`
- `AuthMiddleware.RequireRole("seller", "admin")` — RBAC gate, chained after `Authenticate`
- RBAC roles: `buyer`, `seller`, `admin`
- Auth-service address is configured via `AUTH_SERVICE_ADDR` env var (default `localhost:50051`)

## Build & Run Commands

```bash
# Start infrastructure (PostgreSQL only currently active)
docker compose up -d

# Run auth-service (entrypoint is under cmd/)
cd services/auth-service && go run ./cmd/...

# Run product-catalog-service (entrypoint is main.go at root)
cd services/product-catalog-service && go run .

# Run tests for a service
cd services/<service-name> && go test ./...

# Sync Go workspace
go work sync

# Regenerate Swagger docs
swag init -g cmd/main.go   # auth-service
swag init -g main.go       # product-catalog-service

# Regenerate proto stubs (run from the service's proto/ directory)
protoc --go_out=. --go-grpc_out=. <file>.proto
```

## Coding Conventions

- **No ORM** — use `database/sql` with `lib/pq` directly
- **Config** — each service has its own `pkg/config/config.go` reading from environment variables; defaults are development-safe
- **Proto stubs** — generated files live in `proto/<name>pb/` inside the consuming service; they are not imported as a shared module
- **Transactional outbox** — Order/Payment write events to an `outbox` table in the same DB transaction; Debezium CDC publishes to Kafka
- **Idempotency keys** — Order/Payment endpoints use `Idempotency-Key` header cached in Redis for 24 h
- **Atomic inventory** — Redis Lua scripts do check-and-decrement; PostgreSQL is the durable ledger
