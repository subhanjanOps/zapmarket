## Architecture

ZapMarket is a Go monorepo of six microservices managed via `go.work`. Each service is an independent Go module under `services/`. Shared packages live in `pkg/`.

| `notification-service` | â€” | â€” | â€” |

**Communication:**
- **gRPC** for synchronous inter-service calls (e.g., product-catalog-service calls auth-service to validate tokens)
- **Kafka** for async events (`order.created`, `payment.processed`, `inventory.reserved`, `user.registered`, etc.)
- **No cross-service DB access** â€” all cross-service data goes through gRPC or Kafka

**Infrastructure** (via `docker-compose.yml`): Only PostgreSQL is currently active. Kafka, Redis, Elasticsearch, MinIO, and Debezium configs exist but are commented out.

## Service Internal Structure

Each service follows this layered pattern:
```
internal/
  domain/       # models and domain types
  repository/   # DB access (raw sql.DB, no ORM)
  service/      # business logic; repository_interfaces.go defines the contracts
  handler/
    http/       # HTTP handlers; base.go has shared helpers (ErrorResponse, etc.)
    grpc/       # gRPC server implementation
  middleware/   # auth middleware (JWT validation via gRPC to auth-service)
  errors/       # domain-specific error types
pkg/config/     # service-specific config loaded from env vars
proto/          # .proto files + generated *pb.go files (copied locally per service)
docs/           # Swagger docs (auto-generated by swaggo/swag)
```

## Auth Flow

`auth-service` issues JWTs and exposes a `ValidateToken` gRPC RPC. Other services (e.g., product-catalog-service) call this RPC via `internal/middleware/auth.go` (`AuthMiddleware`) instead of validating JWTs locally. The middleware extracts the `Authorization: Bearer <token>` header and forwards it to auth-service.

RBAC roles: `buyer`, `seller`, `admin`. The `RequireRole` middleware helper is chained after `Authenticate`.

## Commands

**Start infrastructure:**
```bash
docker compose up -d
```

**Run a service:**
```bash
cd services/auth-service
go run ./cmd/...     # auth-service entrypoint is under cmd/
# or
cd services/product-catalog-service
go run .             # entrypoint is main.go at root
```

**Sync workspace:**
```bash
go work sync
```

**Regenerate Swagger docs** (from within the service directory):
```bash
swag init
# auth-service: swag init -g cmd/main.go
# product-catalog-service: swag init -g main.go
```

**Regenerate protobufs** (from service's proto/ dir):
```bash
protoc --go_out=. --go-grpc_out=. <file>.proto
```

**Run tests:**
```bash
cd services/<service-name>
go test ./...
```

## Configuration

All config is loaded from environment variables with sensible defaults (see `pkg/config/config.go` and each service's `pkg/config/config.go`). Both implemented services load a `.env` file via `godotenv` if present â€” see each service's `.env.example` for all available vars. Key env vars:

- `DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD`, `DB_NAME`
- `JWT_SECRET_KEY`, `JWT_REFRESH_SECRET_KEY`
- `HTTP_PORT`, `GRPC_PORT`
- `AUTH_SERVICE_ADDR` (product-catalog-service: address of auth-service gRPC, default `localhost:50051`)

Default DB credentials: user=`zapuser`, password=`zappass123`. PostgreSQL must have `pgcrypto` enabled (for `gen_random_uuid()`).

## Key Design Rules

- **No ORM** â€” repositories use `database/sql` directly with `lib/pq`
- **Transactional outbox** â€” Order and Payment services write events to an `outbox` table in the same transaction, then Debezium CDC publishes to Kafka
- **Idempotency keys** â€” Order and Payment endpoints use `Idempotency-Key` header; result cached in Redis for 24h
- **Atomic inventory** â€” Redis Lua scripts do check-and-decrement to prevent oversell; PostgreSQL is the durable ledger
- `auth-service` uses the standard `net/http` ServeMux; `product-catalog-service` uses `go-chi/chi/v5` with explicit middleware chaining
- Proto-generated files are copied into each consuming service under `proto/<name>pb/` rather than imported from a shared module
